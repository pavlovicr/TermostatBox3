 Test app accommodates two types of tests\+: \`{}functionality test\`{} and \`{}benchmark test\`{}. Both tests are provided per each function written in assembly (typically per each assembly file). Both test apps use a hard copy of LVGL blending API, representing an ANSI implementation of the LVGL blending functions. The hard copy is present in \href{main/lv_blend/}{\texttt{ {\ttfamily lv\+\_\+blend}}} folder.

Assembly source files could be found in the \href{../../src/lvgl9/simd/}{\texttt{ {\ttfamily lvgl\+\_\+port}}} component. \item Tests, whether the HW accelerated assembly version of an LVGL function provides the same results as the ANSI version
\item A top-\/level flow of the functionality test\+:
\begin{DoxyItemize}
\item generate a test matrix with test parameters (matrix width, matrix height, memory alignment.. )
\item run an ANSI version of a DUT function with the generated input parameters
\item run an assembly version of a DUT function with the same input parameters
\item compare the results given by the ANSI and the assembly DUTs
\item the results shall be the same
\item repeat all the steps for a set of different input parameters, checking different matrix heights, widths..
\end{DoxyItemize}

\begin{DoxyItemize}
\item Tests, whether the HW accelerated assembly version of an LVGL function provides a performance increase over the ANSI version
\item A top-\/level flow of the functionality test\+:
\begin{DoxyItemize}
\item generate a test matrix with test parameters (matrix width, matrix height, memory alignment.. )
\item run an ANSI version of a DUT function with the generated input parameters multiple times (1000 times for example), while counting CPU cycles
\item run an assembly version of a DUT function with the generated input parameters multiple times (1000 times for example), while counting CPU cycles
\item compare the results given by the ANSI and the assembly DUTs
\item the assembly version of the DUT function shall be faster than the ANSI version of the DUT function
\end{DoxyItemize}

The test provides couple of information\+:
\begin{DoxyItemize}
\item Total number of CPU cycles for the whole DUT function
\begin{DoxyItemize}
\item {\ttfamily 5363.\+123} cycles for the assembly DUT function
\item {\ttfamily 26219.\+137} cycles for the ANSI DUT function
\end{DoxyItemize}
\item Number of CPU cycles per sample, which is basically the total number of CPU cycles divided by the test matrix area
\begin{DoxyItemize}
\item {\ttfamily 0.\+327} cycles per sample for the assembly DUT
\item {\ttfamily 1.\+6} cycles per sample for the ANSI DUT
\item In this case, the assembly implementation has achieved a performance increase in around 4.\+9-\/times, comparing to the ANSI implementation.
\end{DoxyItemize}
\item Range of the CPU cycles (a best case and a corner case scenarios) into which, the DUT functions are expected to fit into
\begin{DoxyItemize}
\item The execution time of those function highly depends on the input parameters, thus a boundary scenarios for input parameters shall be set
\item An example of such a boundaries is in a table below
\item The benchmark boundary would help us to get an performance expectations of the real scenarios
\end{DoxyItemize}
