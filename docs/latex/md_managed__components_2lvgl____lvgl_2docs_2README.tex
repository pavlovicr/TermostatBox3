\item Doxygen
\item Python \texorpdfstring{$>$}{>}= 3.\+10
\item C compiler (gcc, msvc, clang, etc...)
\end{DoxyItemize}

Once Python is installed \begin{DoxyVerb}pip install -r requirements.txt
\end{DoxyVerb}
 will install all the prerequisite packages\+:

\item Sphinx
\item breathe
\item imagesize
\item importlib-\/metadata
\item sphinx-\/rtd-\/theme
\item sphinx-\/sitemap
\item sphinxcontrib-\/applehelp
\item sphinxcontrib-\/devhelp
\item sphinxcontrib-\/htmlhelp
\item sphinxcontrib-\/jsmath

\item sphinx-\/design
\item typing-\/extensions

\item dirsync
\item furo
\item accessible-\/pygments

Now you are ready to build the documentation\+: \begin{DoxyVerb}python build.py html
\end{DoxyVerb}
 or if you are on a Unix like OS\+: \begin{DoxyVerb}python3 build.py html
\end{DoxyVerb}
 Intermediate files are normally prepared in {\ttfamily ./docs/intermediate/} and the final documentation will normally appear in {\ttfamily ./docs/build/html/}. (Both of these directories can be overridden using environment variables. See documentation in {\ttfamily build.\+py} header comment for details.)

If the list of document source files has changed (names or paths)\+: \begin{DoxyVerb}python build.py clean html
\end{DoxyVerb}
 Will remove the old intermediate and build files and regenerate new ones matching the new structure, eliminating the orphan files that would otherwise result.

To see a list of options available\+: \begin{DoxyVerb}python build.py
\end{DoxyVerb}

One of our firm policies is {\itshape {\bfseries{EVERYTHING MUST BE DOCUMENTED}}}.

The below are some rules to follow when updating any of the {\ttfamily .rst} files located in the {\ttfamily ./docs/src/} directory tree.\hypertarget{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md424}{}\doxysubsection{\texorpdfstring{What to Name Your $<$tt$>$.\+rst$<$/tt$>$ File}{What to Name Your <tt>.rst</tt> File}}\label{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md424}
The directory structure under the {\ttfamily ./docs/src/} directory, and the filenames of the {\ttfamily .rst} files govern the eventual URLs that are generated in the HTML output. These directories are organized so as to reflect the nature of the content. Example\+: the {\ttfamily .rst} files under {\ttfamily ./docs/src/intro} contain introductory material—detailed reference material would not go there, but instead in an appropriate subdirectory of {\ttfamily ./docs/src/details/}. It is expected that the content and location of any new documents added would be in alignment with this directory structure, and placed and named according to their content. Additionally, to be linked into the eventual generated documentation, the stem of the new filename needs to appear in at least one (normally {\itshape only one}) {\ttfamily .. toctree\+:\+:} directive, normally in an {\ttfamily index.\+rst} file in the directory where it will appear in that page\textquotesingle{}s table of contents (TOC).

Other than that, there are no restrictions on filenames. Previous linking of filenames to generated API links has been removed and replaced by a better scheme. For sake of illustration, let\textquotesingle{}s say you are creating (or enhancing) documentation related to the {\ttfamily lv\+\_\+scale\+\_\+t} data type (one of the LVGL Widgets)\+: if you want the doc-\/build logic to generate appropriate links to LVGL API pages, place an API section at the end of your document (it must be at the end) like this\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{API}
\DoxyCodeLine{***}

\end{DoxyCode}


and then, if you want the API-\/link-\/generation logic to generate hyperlinks to API pages based on an {\itshape {\bfseries{exact, case-\/sensitive string match}}} with specific C symbols, follow it with a re\+Structured\+Text comment using this syntax\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{..\ API\ equals:\ lv\_scale\_t,\ lv\_scale\_create}

\end{DoxyCode}


What follows the colon is a comma-\/ or space-\/separated list of exact C symbols documented somewhere in the {\ttfamily lvgl/src/} directory. If the list is long, it can be wrapped to subsequent lines, though continuation lines must be all indented at the same level. The list ends with the first blank line after this pseudo-\/directive.

If you instead want the API-\/link-\/generation logic to simply include links to code that {\itshape {\bfseries{starts with a specific string}}} use this syntax instead. The format of the list is the same as for {\ttfamily .. API equals\+:}\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{..\ API\ startswith:\ lv\_scale,\ lv\_obj\_set\_style}

\end{DoxyCode}


You can also manually link to API pages, in which case the API-\/link-\/generation logic will see that you have already added links and will not repeat them.


\begin{DoxyCode}{0}
\DoxyCodeLine{:ref:`lv\_scale\_h`}

\end{DoxyCode}


Note that the period before the {\ttfamily h} is replaced with an underscore ({\ttfamily \+\_\+}). The naming of this reference ({\ttfamily lv\+\_\+scale\+\_\+h}) will generate a hyperlink to the documentation extracted by Doxygen from the {\ttfamily \doxylink{lv__scale_8h}{lvgl/src/widgets/scale/lv\+\_\+scale.\+h}} file.\hypertarget{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md425}{}\doxysubsection{\texorpdfstring{Text Format}{Text Format}}\label{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md425}
With {\ttfamily .md} files, it is important to allow paragraphs to flow off to the right with one long line so that when they are formatted as {\ttfamily .html} files, the paragraphs will word-\/wrap with the width of the browser. Thankfully, this liability is not present with re\+Structured\+Text ({\ttfamily .rst} files). \href{https://www.sphinx-doc.org/en/master/}{\texttt{ Sphinx}} and its underlying \href{https://docutils.sourceforge.io/docs/}{\texttt{ docutils parsing engine}} conveniently combine grouped text into a proper paragraph with that word-\/wrapping behavior. This allows the source text documents to be nicely word-\/wrapped so that they are more readable in text-\/ and code-\/editors that do not have wide editing windows. So please wrap the text around column 86 or narrower. Wrapping at {\itshape exactly} column 86 is not important, but readability and ease of editing is.\hypertarget{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md426}{}\doxysubsection{\texorpdfstring{index.\+rst Files}{index.rst Files}}\label{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md426}
If you create a new directory you will need an {\ttfamily index.\+rst} file in that directory and that index file needs to be pointed to in the {\ttfamily index.\+rst} file that is located in the parent directory.

The first line is for the purposes of providing a uniquely-\/named {\bfseries{link target}} that can be referenced elsewhere in the documentation. \begin{DoxyVerb}.. _{LINK NAME}:
\end{DoxyVerb}
 Note that {\ttfamily \{LINK NAME\}}\+:


\begin{DoxyItemize}
\item {\bfseries{must}} be preceded by a single underscore,
\item {\bfseries{must}} be followed by a single colon, and
\item {\bfseries{must}} be followed by at least one blank line for the doc-\/generation logic to process it correctly.
\end{DoxyItemize}

Replace {\ttfamily \{LINK NAME\}} with a link name that is unique among all documents under the {\ttfamily ./docs/src/} directory. It can have multiple words if needed to make it unique or when otherwise appropriate for clarity. If multiple words are used, they can be separated with single spaces, hyphens or underscores. Whatever you use, the {\ttfamily \{LINK NAME\}} string used to reference it must be identical. {\ttfamily \{LINK NAME\}} strings are not case sensitive.

That unique name is then used to provide a link reference elsewhere in the documentation using one of two formats.\hypertarget{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md427}{}\doxyparagraph{\texorpdfstring{When the "{}link text"{} should be a title or section heading from the target document\+:}{When the "link text" should be a title or section heading from the target document:}}\label{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md427}

\begin{DoxyCode}{0}
\DoxyCodeLine{:ref:`\{LINK\ NAME\}`}

\end{DoxyCode}


This in-\/line markup (interpreted text using the Sphinx-\/defined custom {\ttfamily \+:ref\+:} role) is then replaced with a hyperlink whose "{}link text"{} is the name of the section heading or document title just below the {\bfseries{link target}}. For this reason, when using this syntax, {\ttfamily \{LINK NAME\}} must reference \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}link target\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}s that are just above a title or section heading.\hypertarget{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md428}{}\doxyparagraph{\texorpdfstring{When "{}link text"{} should be something else\+:}{When "link text" should be something else:}}\label{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md428}

\begin{DoxyCode}{0}
\DoxyCodeLine{:ref:`other\ link\ text\ <\{LINK\ NAME\}>`}

\end{DoxyCode}


This latter syntax enables you to put a {\bfseries{link target}} anywhere in an .RST file (not just above a heading) and link to it using this syntax.

Note\+: This latter syntax was either added or fixed in Sphinx recently. It did not work in Sphinx 7.\+3.\+7.\hypertarget{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md429}{}\doxysubsection{\texorpdfstring{Section Headings}{Section Headings}}\label{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md429}
\href{https://www.sphinx-doc.org/en/master/usage/restructuredtext/basics.html\#sections}{\texttt{ Section headings}} are created by underlining the section title with a punctuation character, at least as long as the text. Example\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{This\ Is\ a\ Heading}
\DoxyCodeLine{*****************}

\end{DoxyCode}


re\+Structured\+Text does not impose any particular heading levels assigned to certain characters since the structure is determined from the succession of headings. However, the LVGL docs policy is to use the below heading convention\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{=====}
\DoxyCodeLine{Title}
\DoxyCodeLine{=====}
\DoxyCodeLine{}
\DoxyCodeLine{Chapter}
\DoxyCodeLine{*******}
\DoxyCodeLine{}
\DoxyCodeLine{Section}
\DoxyCodeLine{-\/-\/-\/-\/-\/-\/-\/}
\DoxyCodeLine{}
\DoxyCodeLine{Sub\ Section}
\DoxyCodeLine{\string~\string~\string~\string~\string~\string~\string~\string~\string~\string~\string~}
\DoxyCodeLine{}
\DoxyCodeLine{Sub\ Sub\ Section}
\DoxyCodeLine{\string^\string^\string^\string^\string^\string^\string^\string^\string^\string^\string^\string^\string^\string^\string^}
\DoxyCodeLine{}
\DoxyCodeLine{Sub\ Sub\ Sub\ Section}
\DoxyCodeLine{'''''''''''''''''''}

\end{DoxyCode}


Being consistent about this helps the Sphinx/docutils parser to format the tables of contents correctly.

Note that the "{}underlining"{} can be longer than the heading title, but if it is shorter, the documentation-\/generation logic will fail with an error.

For improved readability in the .RST file\+:


\begin{DoxyItemize}
\item place 3 blank lines above the 2nd and subsequent chapter titles (see above), and
\item place 2 blank lines above section headings below chapters.
\end{DoxyItemize}\hypertarget{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md430}{}\doxysubsection{\texorpdfstring{Code Blocks}{Code Blocks}}\label{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md430}

\begin{DoxyItemize}
\item Do not use tab characters in code blocks.
\item Each indentation level use 4 spaces.
\item Include at least 1 empty line after a code block.
\item There must be one empty line between the code block directive and the code.
\item {\ttfamily .. code-\/block\+::} is the only directive that should be used. Note carefully that unlike the {\bfseries{link target}} directive above, this directive has 2 colons. (The only re\+ST and Sphinx directives that are valid with one colon are \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}link target\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}s as shown above.) Lone {\ttfamily \+::}, {\ttfamily \+:code\+:} or {\ttfamily .. code\+:} should not be used.
\item If you want to separate code into easier-\/to-\/understand sections you can do so with a single empty line.
\item For syntax highlighting appropriate to the language in the code block, specify the language after the directive. Some examples are\+:
\begin{DoxyItemize}
\item {\ttfamily .. code-\/block\+:: c},
\item {\ttfamily .. code-\/block\+:: cpp},
\item {\ttfamily .. code-\/block\+:: python},
\item {\ttfamily .. code-\/block\+:: shell},
\item {\ttfamily .. code-\/block\+:: bash},
\item {\ttfamily .. code-\/block\+:: kconfig},
\item {\ttfamily .. code-\/block\+:: json},
\item {\ttfamily .. code-\/block\+:: yaml},
\item {\ttfamily .. code-\/block\+:: csharp} (or "{}cs"{}),
\item {\ttfamily .. code-\/block\+:: vb.\+net},
\item {\ttfamily .. code-\/block\+:: dot} (graphviz),
\item {\ttfamily .. code-\/block\+:: html},
\item {\ttfamily .. code-\/block\+:: css},
\item {\ttfamily .. code-\/block\+:: xml},
\item {\ttfamily .. code-\/block\+:: make}.
\end{DoxyItemize}
\end{DoxyItemize}

The full set of supported lexers are listed in the \href{https://pygments.org/docs/lexers/}{\texttt{ Pygments Library documentation}}.\hypertarget{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md431}{}\doxysubsection{\texorpdfstring{Bulleted Lists}{Bulleted Lists}}\label{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md431}
To create a bulleted list, do the following\+: \begin{DoxyVerb}- First item description
- If you want to span multiple lines, indent subsequent
  lines to align with item text like this.
- If you want to include a code block under a list item,
  it must be intended to align with the list item like this:

  .. code-block: python
                             <=== blank line here is important
      # this is some code
                             <=== blank line here is important
- If you want to have nested bulleted lists, indent each
  new level to align with its parent list item like this:
                             <=== blank line here is important
  - level 2 item 1: text
  - level 2 item 2: text
                             <=== blank line here is important
- Last list item.  Note that the nested list above is preceded
  and followed by 1 blank line.
\end{DoxyVerb}
 All lists (including nested lists) {\bfseries{must}} be preceded and followed with at least 1 blank line. This is mandatory for the documentation-\/generation logic to process it correctly.\hypertarget{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md432}{}\doxysubsection{\texorpdfstring{Referencing API Documentation}{Referencing API Documentation}}\label{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md432}
If you want to reference portions of the LVGL code from the documentation (in .RST files) there are special directives to do this\+: \begin{DoxyVerb}:cpp:func:`lv_init`
:c:macro:`LV_USE_FLEX`
:cpp:type:`lv_event_t`
:cpp:enum:`lv_state_t`
:cpp:enumerator:`LV_STATE_CHECKED`
:cpp:struct:`lv_image_dsc_t`
:cpp:union:`lv_style_value_t`
\end{DoxyVerb}
 There is a special directive when wanting to use a more complex expression. For example when showing the arguments passed to a function. \begin{DoxyVerb}:cpp:expr:`lv_obj_set_layout(widget, LV_LAYOUT_FLEX)`
:cpp:expr:`lv_slider_set_mode(slider, LV_SLIDER_MODE_...)`
\end{DoxyVerb}
 Arguments that are expressions (more than one word), or contain non-\/alphanumeric characters will cause the {\ttfamily \+:cpp\+:expr\+:} interpreted-\/text to fail. Examples\+: \begin{DoxyVerb}:cpp:expr:`lv_obj_set_layout(widget, LV_LAYOUT_FLEX/GRID)`         <== arg with > 1 word
:cpp:expr:`lv_obj_set_layout(widget, LV_LAYOUT_*)`                 <== asterisk
:cpp:expr:`lv_obj_set_layout(*widget, LV_LAYOUT_FLEX)`             <== asterisk
:cpp:expr:`lv_obj_set_layout((lv_obj_t *)widget, LV_LAYOUT_FLEX)`  <== cast
:cpp:expr:`lv_obj_set_layout(&widget, LV_LAYOUT_FLEX);`            <== ampersand & semicolon
:cpp:expr:`lv_obj_set_layout(widget, ...)`                         <== lone ellipsis
\end{DoxyVerb}
 For such examples, simply use re\+Structured\+Text literal markup like this\+: \begin{DoxyVerb}``lv_obj_set_layout(widget, LV_LAYOUT_FLEX/GRID)``
``lv_obj_set_layout(widget, LV_LAYOUT_*)``
``lv_obj_set_layout(*widget, LV_LAYOUT_FLEX)``
``lv_obj_set_layout((lv_obj_t *)widget, LV_LAYOUT_FLEX)``
``lv_obj_set_layout(&widget, LV_LAYOUT_FLEX);``
``lv_obj_set_layout(widget, ...)``
\end{DoxyVerb}
