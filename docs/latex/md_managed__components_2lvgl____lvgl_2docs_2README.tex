Building the documentation is easy. Here are the requirements\+:


\begin{DoxyItemize}
\item Doxygen
\item Python \texorpdfstring{$>$}{>}= 3.\+10
\item C compiler (gcc, msvc, clang, etc...)
\end{DoxyItemize}

Once Python is installed \begin{DoxyVerb}pip install -r requirements.txt
\end{DoxyVerb}
 will install all the prerequisite packages.

To build the documentation on Windows\+: \begin{DoxyVerb}python build.py html
\end{DoxyVerb}
 On Linux\+: \begin{DoxyVerb}python3 build.py html
\end{DoxyVerb}
 Intermediate files are normally prepared in {\ttfamily ./docs/intermediate/} and the final documentation will normally appear in {\ttfamily ./docs/build/html/}. Both of these directories can be overridden using environment variables. See documentation in {\ttfamily build.\+py} header comment for details.

If the list of document source files (including the {\ttfamily .h} files in the {\ttfamily ./src/} directory) has changed (names or paths)\+: \begin{DoxyVerb}python build.py clean
\end{DoxyVerb}
 Will remove the old intermediate and build files, eliminating the orphan files that would otherwise result.

To see a list of options available\+: \begin{DoxyVerb}python build.py
\end{DoxyVerb}
 Read the header comment in {\ttfamily build.\+py} for detailed documentation on each option.\hypertarget{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md490}{}\doxysection{\texorpdfstring{For Developers}{For Developers}}\label{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md490}

Section headings are created by underlining the section title with a punctuation character, at least as long as the text. Example\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{This\ Is\ a\ Heading}
\DoxyCodeLine{*****************}

\end{DoxyCode}


Use these conventions for section headings in LVGL documentation\+:

\DoxyCodeLine{==============}
\DoxyCodeLine{Document\ Title}
\DoxyCodeLine{==============}
\DoxyCodeLine{}
\DoxyCodeLine{Section}
\DoxyCodeLine{*******}
\DoxyCodeLine{}
\DoxyCodeLine{Sub\ Section}
\DoxyCodeLine{-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/}
\DoxyCodeLine{}
\DoxyCodeLine{Sub\ Sub\ Section}
\DoxyCodeLine{\string~\string~\string~\string~\string~\string~\string~\string~\string~\string~\string~\string~\string~\string~\string~}

\end{DoxyCode}



\begin{DoxyEnumerate}
\item {\ttfamily ====}, {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}}, {\ttfamily -\/-\/-\/-\/} are encouraged.
\item {\ttfamily \texorpdfstring{$\sim$}{\string~}\texorpdfstring{$\sim$}{\string~}\texorpdfstring{$\sim$}{\string~}\texorpdfstring{$\sim$}{\string~}} only if really needed.
\end{DoxyEnumerate}

Being consistent about this helps the re\+ST parser to format the tables of contents correctly.

For improved readability in text editors\+:


\begin{DoxyItemize}
\item place 3 blank lines above the 2nd and subsequent "{}\+Section"{} titles (see above),
\item 2 blank lines above "{}\+Sub Section"{} headings, and
\item 1 at least blank line above all lower-\/level section headings.
\end{DoxyItemize}\hypertarget{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md495}{}\doxysubsection{\texorpdfstring{Italics, Boldface and Underlining}{Italics, Boldface and Underlining}}\label{md_managed__components_2lvgl____lvgl_2docs_2README_autotoc_md495}
Emphasis using {\ttfamily \texorpdfstring{$\ast$}{*}italics\texorpdfstring{$\ast$}{*}}. Emphasis using {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}boldface\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}}.

Normally underlining and combining these text styles is not possible in re\+Structured\+Text. However, LVGL documentation provides a work-\/around for this using re\+ST \uline{Interpreted Text Roles}. Just remember that the Interpreted Text Role names combine the letters {\ttfamily i}, {\ttfamily b} and {\ttfamily u} to provide the desired combination. All possible permutations of these letters are supported so you do not have to remember what sequence works. Because not everyone has editors that deal with Unicode characters well, please re\+ST substitutions to insert special characters into documentation. A list of the most commonly-\/used special symbols can be found in {\ttfamily ./docs/src/include/substitutions.txt}. To use one of these, add this line at the top of the {\ttfamily .rst} file if it is not already there\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{..\ include::\ /include/substitutions.txt}

\end{DoxyCode}


Then, any of those substitutions can be used in that {\ttfamily .rst} file. Example\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{The\ temperature\ outside\ is\ 20\(\backslash\)\ |deg|\(\backslash\)\ C.}

\end{DoxyCode}


Result\+: "{}\+The temperature outside is 20°\+C."{}

The spaces surrounding substitutions {\itshape are required for parsing}, but when you need to remove them in the output (as in the example above), do so by escaping them with the {\ttfamily \textbackslash{}} character. Exception\+: the {\ttfamily substitutions.\+txt} file contains 3 substitution definitions which are marked with the {\ttfamily \+:trim\+:} option since their use {\itshape always} removes these spaces in the output. These do not need this escaping\+:


\begin{DoxyItemize}
\item {\ttfamily \texorpdfstring{$\vert$}{|}nbsp\texorpdfstring{$\vert$}{|}} (non-\/breaking space),
\item {\ttfamily \texorpdfstring{$\vert$}{|}shy\texorpdfstring{$\vert$}{|}} (soft hyphen), and
\item {\ttfamily \texorpdfstring{$\vert$}{|}nbhyph\texorpdfstring{$\vert$}{|}} (non-\/breaking hyphen used in titles and official names)
\end{DoxyItemize}

To create a link to 1 or more API pages, set up a section at the end of your {\ttfamily .rst} file that looks like this, and use one or both types of the {\ttfamily .. API} pseudo-\/directives below\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{API}
\DoxyCodeLine{***}
\DoxyCodeLine{}
\DoxyCodeLine{..\ API\ equals:\ lv\_scale\_t,\ lv\_scale\_create}
\DoxyCodeLine{}
\DoxyCodeLine{..\ API\ startswith:}
\DoxyCodeLine{}
\DoxyCodeLine{\ \ \ \ lv\_scale}
\DoxyCodeLine{\ \ \ \ lv\_obj\_set\_style}

\end{DoxyCode}


The list of symbols (or prefixes) can be separated by commas or spaces, and they can wrap onto subsequent lines of text so long as they are indented. Each list is terminated by the next {\ttfamily .. API} pseudo-\/directive or end-\/of-\/file, whichever comes first.

The API-\/page generation logic will add at most 1 link to each API documentation page containing matched symbols. The links are to the whole API page, not to the symbols. The purpose is to provide the reader with links to applicable API pages. Links directly to code (e.\+g. function documentation) are accomplished by using the In-\/\+Line Code Expressions documented above.
