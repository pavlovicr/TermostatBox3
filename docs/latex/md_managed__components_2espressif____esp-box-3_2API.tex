\hypertarget{md_managed__components_2espressif____esp-box-3_2API_autotoc_md104}{}\doxysection{\texorpdfstring{Overview}{Overview}}\label{md_managed__components_2espressif____esp-box-3_2API_autotoc_md104}
This document provides an overview of the ESP-\/\+BSP (Board Support Package) API as implemented by this board.

While the ESP-\/\+BSP framework defines a unified API shared across multiple boards, this documentation focuses only on the APIs supported by the current board. Any APIs not applicable to this board\textquotesingle{}s hardware are excluded or may not be functional.

The goal of this document is to make it easier for developers to understand the available APIs and how to use them consistently across different boards.\hypertarget{md_managed__components_2espressif____esp-box-3_2API_autotoc_md105}{}\doxysection{\texorpdfstring{General}{General}}\label{md_managed__components_2espressif____esp-box-3_2API_autotoc_md105}
\hypertarget{md_managed__components_2espressif____esp-box-3_2API_autotoc_md106}{}\doxysubsection{\texorpdfstring{Pinout}{Pinout}}\label{md_managed__components_2espressif____esp-box-3_2API_autotoc_md106}
Each BSP defines a set of macros for default pin assignments used by its hardware peripherals. These macros allow users to configure or reference standard interfaces like I2C, SPI, LCD, audio, or SD cards easily.


\begin{DoxyItemize}
\item I2C\+: {\ttfamily BSP\+\_\+\+I2\+C\+\_\+\texorpdfstring{$\ast$}{*}}
\item Display\+: {\ttfamily BSP\+\_\+\+LCD\+\_\+\texorpdfstring{$\ast$}{*}}
\item Audio I2S\+: {\ttfamily BSP\+\_\+\+I2\+S\+\_\+\texorpdfstring{$\ast$}{*}}
\item USB\+: {\ttfamily BSP\+\_\+\+USB\+\_\+\texorpdfstring{$\ast$}{*}}
\item SD Card (MMC)\+: {\ttfamily BSP\+\_\+\+SD\+\_\+\texorpdfstring{$\ast$}{*}}
\item SD Card (SPI)\+: {\ttfamily BSP\+\_\+\+SD\+\_\+\+SPI\+\_\+\texorpdfstring{$\ast$}{*}}
\end{DoxyItemize}

\begin{quote}
\mbox{[}!\+NOTE\mbox{]} Not all boards support all interfaces. You should always check if the related capability macro (e.\+g., BSP\+\_\+\+CAPS\+\_\+\+SDCARD) is defined. \end{quote}
\hypertarget{md_managed__components_2espressif____esp-box-3_2API_autotoc_md107}{}\doxysubsection{\texorpdfstring{I2C}{I2C}}\label{md_managed__components_2espressif____esp-box-3_2API_autotoc_md107}
Some devices included in BSPs (e.\+g., sensors, displays, audio codecs) communicate via the I2C interface. In many cases, I2C is initialized automatically as part of the device setup. However, you can manually initialize or deinitialize the I2C peripheral using the following API\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{/*\ Initialize\ the\ default\ I2C\ bus\ used\ by\ the\ BSP\ */}
\DoxyCodeLine{bsp\_i2c\_init();}
\DoxyCodeLine{}
\DoxyCodeLine{...}
\DoxyCodeLine{}
\DoxyCodeLine{/*\ Deinitialize\ the\ I2C\ bus\ */}
\DoxyCodeLine{bsp\_i2c\_deinit();}

\end{DoxyCode}


If you need direct access to the initialized I2C bus (e.\+g., to communicate with an external peripheral not handled by the BSP), you can retrieve the I2C bus handle\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{i2c\_master\_bus\_handle\_t\ i2c\ =\ bsp\_i2c\_get\_handle();}

\end{DoxyCode}


\begin{quote}
\mbox{[}!\+NOTE\mbox{]} The BSP ensures that I2C initialization is performed only once, even if called multiple times. This helps avoid conflicts when multiple components rely on the same I2C bus. \end{quote}
\hypertarget{md_managed__components_2espressif____esp-box-3_2API_autotoc_md108}{}\doxysubsection{\texorpdfstring{ADC}{ADC}}\label{md_managed__components_2espressif____esp-box-3_2API_autotoc_md108}
Some devices included in BSPs (such as buttons, battery monitoring, etc.) use the ADC peripheral. In most cases, the ADC is automatically initialized as part of the specific device setup. However, you can manually initialize the ADC using the following API\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{/*\ Initialize\ the\ ADC\ peripheral\ */}
\DoxyCodeLine{bsp\_adc\_initialize();}

\end{DoxyCode}


If you need direct access to the ADC instance (e.\+g., for custom measurements), you can retrieve the handle\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{adc\_oneshot\_unit\_handle\_t\ adc\ =\ bsp\_adc\_get\_handle();}

\end{DoxyCode}


\begin{quote}
\mbox{[}!\+NOTE\mbox{]} The BSP ensures the ADC is initialized only once, even if {\ttfamily bsp\+\_\+adc\+\_\+initialize()} is called multiple times. \end{quote}
\hypertarget{md_managed__components_2espressif____esp-box-3_2API_autotoc_md109}{}\doxysubsection{\texorpdfstring{Features}{Features}}\label{md_managed__components_2espressif____esp-box-3_2API_autotoc_md109}
Some boards support enabling or disabling specific hardware features (such as LCD, SD card, camera, etc.) to reduce power consumption or manage shared resources. The BSP provides a unified API to control these features\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{/*\ Enable\ the\ LCD\ feature\ */}
\DoxyCodeLine{bsp\_feature\_enable(BSP\_FEATURE\_LCD,\ true);}
\DoxyCodeLine{}
\DoxyCodeLine{/*\ Disable\ the\ speaker\ to\ reduce\ power\ usage\ */}
\DoxyCodeLine{bsp\_feature\_enable(BSP\_FEATURE\_SPEAKER,\ false);}

\end{DoxyCode}


Supported feature flags (may vary depending on the board)\+:
\begin{DoxyItemize}
\item {\ttfamily BSP\+\_\+\+FEATURE\+\_\+\+LCD} -\/ Display module
\item {\ttfamily BSP\+\_\+\+FEATURE\+\_\+\+TOUCH} -\/ Touch controller
\item {\ttfamily BSP\+\_\+\+FEATURE\+\_\+\+SD} -\/ SD card interface
\item {\ttfamily BSP\+\_\+\+FEATURE\+\_\+\+SPEAKER}-\/ Audio speaker
\item {\ttfamily BSP\+\_\+\+FEATURE\+\_\+\+BATTERY} -\/ Battery monitoring
\item {\ttfamily BSP\+\_\+\+FEATURE\+\_\+\+VIBRATION} -\/ Vibration motor
\end{DoxyItemize}

\begin{quote}
\mbox{[}!\+NOTE\mbox{]} Not all BSPs support feature toggling, and some features may not be available or controllable via this API. Always check the BSP header or documentation for supported features. \end{quote}
\begin{quote}
\mbox{[}!\+TIP\mbox{]} Disabling unused features can help reduce power consumption, especially in battery-\/powered applications. \end{quote}
